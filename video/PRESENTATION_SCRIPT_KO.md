## 3page — 프로젝트 배경

먼저 이 프로젝트를 시작하게 된 배경에 대해 말씀드리겠습니다.

국내 요가 인구는 꾸준히 증가하고 있지만, 수련자들은 자신의 부상 이력과 신체 조건에 맞는 포즈를 찾고 적합한 강사를 연결받는 데 큰 어려움을 겪고 있습니다. 특히 허리 디스크, 무릎 부상, 고혈압 등 의료적 주의가 필요한 수련자의 경우, 검증되지 않은 콘텐츠를 무방비로 따라 하다가 부상이 악화되는 사례가 실제로 발생하고 있습니다.

한편 인스타그램에는 고품질 강사 프로필이 넘쳐나, 수련자 입장에서는 단편적인 게시물만으로 강사의 전문성과 자격을 실질적으로 구별하기가 매우 어렵습니다. 20년 이상의 경력을 보유한 전문 강사도 파편화된 SNS 정보 속에 묻혀 제대로 된 신뢰 신호를 전달하지 못하고 있으며, AI 시대의 검색·추천 흐름에서 지속적으로 이탈되고 있습니다.

이 두 가지 문제, 즉 **수련자의 안전 공백**과 **강사의 디지털 가시성 부재**를 동시에 해결하는 신뢰 기반 AI 매칭 플랫폼이 필요하다고 판단하여 요가큐를 시작하게 되었습니다.

## 4page — 기존 플랫폼 분석 ①: 검색·소셜 미디어 채널 (네이버·구글·인스타그램·유튜브)

먼저 국내 수련자들이 요가 정보를 가장 많이 탐색하는 경로인 **검색·소셜 미디어 채널**을 살펴보겠습니다.

네이버·구글은 콘텐츠 양과 접근성 면에서 장점이 있지만, 검색 결과 상위를 광고와 일반 블로그가 차지하며 사용자의 신체 조건은 전혀 반영되지 않습니다. 인스타그램과 유튜브는 풍부한 요가 영상 콘텐츠를 제공하지만, 알고리즘이 조회수·팔로워 수 중심으로 콘텐츠를 노출하기 때문에 강사의 실제 전문성이나 자격과 무관하게 인기 있는 콘텐츠가 먼저 추천되는 구조입니다.

특히 허리 디스크 환자가 "초보자 요가"를 검색하거나 관련 영상을 추천받았을 때 오히려 척추 부담이 큰 포즈가 상위에 노출되는 사례처럼, **이 채널들 모두 금기 포즈 필터링 기능이 전혀 없다는 심각한 안전 공백**을 공통적으로 가지고 있습니다.

## 5page — 기존 플랫폼 분석 ②: Mindbody · ClassPass

대표적인 요가 스튜디오 예약 마켓플레이스인 Mindbody와 ClassPass를 살펴보면, 스튜디오 검색과 예약 편의성이 높고 글로벌 네트워크를 보유하고 있다는 장점이 있습니다.

그러나 이 플랫폼들은 단순 예약 인터페이스에 머물러 있어, 수련자의 건강 상태나 목표에 맞는 추천이 전혀 이루어지지 않습니다. 또한 Schema.org 구조화 데이터를 제공하지 않기 때문에 Gemini, SearchGPT, Perplexity 등 AI 검색엔진에서 권위 있는 소스로 인용되지 못하는 구조적 한계도 있습니다.

## 6page — 기존 플랫폼 한계 정리 → 해결 방향

이렇게 주요 플랫폼을 분석한 결과, 기존 서비스의 한계를 크게 다섯 가지로 정리할 수 있었습니다.

포즈가 단순 나열되어 안전성을 사용자가 스스로 판단해야 하고, 강사 자격 정보가 구조화되어 제공되지 않는 문제가 있었습니다. 또한 신체 조건·부상 이력을 반영한 필터링 기능이 부재하고, AI 검색엔진이 인용할 수 있는 Schema.org 구조화 데이터가 없었습니다. 마지막으로 위치 기반 스튜디오 전문성 매칭도 불가능했습니다.

저는 이 다섯 가지 문제를 아래와 같이 해결합니다:

| 문제 | 요가큐 해결 방식 |
|------|---------------------|
| 안전성 판단 불가 | Kill-Switch 금기사항 하드 차단 — 위험 포즈 자동 제외 |
| 강사 자격 불투명 | `instructor_trust_score` + Schema.org Person JSON-LD |
| 개인화 필터 없음 | 경험 레벨 · 가용 시간 · 목표 기반 다중 필터 매칭 |
| AI 인용 불가 | FAQPage · HowTo · DefinedTerm JSON-LD 자동 생성 |
| 위치 매칭 없음 | Haversine 반경 쿼리 + 전문 분야 가중치 랭킹 |

## 7page — 활용 데이터

프로젝트에서 활용한 주요 데이터는 크게 네 가지입니다.

첫째, **포즈 데이터베이스**입니다. 2,700개 이상의 요가 아사나를 기반으로 E-E-A-T 메타데이터(혜택·금기사항·해부학적 초점·난이도)를 자체 enrichment 파이프라인으로 보강하였으며, PostgreSQL에 Flyway 마이그레이션으로 관리하고 있습니다.

둘째, **Q&A 학습 데이터**입니다. GPT-4o-mini를 활용해 856개 포즈에 대한 자연어 설명과 2,899개의 FAQ 쌍을 생성하였습니다. 이 데이터는 AI 검색 인용을 위한 FAQPage JSON-LD로 API에서 직접 제공됩니다.

셋째, **강사·스튜디오 데이터**입니다. 서울 주요 스튜디오와 강사 정보를 구조화하여 위치 기반 매칭 및 Schema.org Person JSON-LD 생성에 활용합니다.

넷째, **교육 자료(OCR 파이프라인)**입니다. 요가 전문 도서를 Tesseract OCR로 추출하고 nomic-embed-text 모델로 벡터 임베딩하여, RAG 기반 챗봇이 책 내용을 직접 인용하며 질문에 답할 수 있도록 하였습니다.

## 8page — 기술 스택

다음은 기술 스택을 소개하겠습니다.

**백엔드**는 Spring Boot 3.x + Java 17로 매칭 엔진을 구축하였고, Spring Data JPA + Flyway로 PostgreSQL 스키마를 버전 관리합니다. RESTful API는 SpringDoc OpenAPI로 자동 문서화됩니다.

**AI 레이어**는 FastAPI(Python) 기반 RAG 파이프라인으로, Ollama의 nomic-embed-text 모델로 벡터 임베딩을 처리하고 mistral:latest 로컬 LLM으로 추론을 수행합니다. 포즈 Q&A 데이터 생성에는 GPT-4o-mini를 활용하였습니다.

**프론트엔드**는 React 18 + Vite 5로 구성하였으며, Streamlit을 이용한 인터랙티브 데모 페이지(매칭 점수 시각화 · 안전 필터 시연)도 별도 제공합니다.

**인프라**는 Docker Compose로 서비스를 컨테이너화하고, Caddy 리버스 프록시와 Cloudflare 터널을 통해 match.yogaman.club에 배포하였습니다.

그 외 사용된 기술 스택은 화면 자료를 참고해 주시기 바랍니다.

## 9page — 주요 기능

다음은 요가큐 서비스의 사용자별 주요 기능을 설명드리겠습니다.

서비스는 **수련자**와 **강사·스튜디오**, 두 사용자 그룹의 니즈에 맞춰 설계되었습니다.

**수련자 기능입니다.**
수련자는 자연어로 신체 조건과 목표를 입력하면 AI 매칭 엔진이 적합한 포즈를 추천합니다. 허리 디스크·무릎 부상·고혈압 등 의료 금기가 있을 경우 Kill-Switch 필터가 위험 포즈를 자동 차단하며, 차단 사유도 함께 제공됩니다. 경험 레벨과 가용 시간(분 단위)으로 세션을 추가 필터링할 수 있고, 위치 기반 스튜디오 검색으로 주변의 전문 강사를 연결받을 수 있습니다. RAG 챗봇을 통해 포즈와 요가 지식에 대한 자유로운 질문도 가능합니다.

**강사·스튜디오 기능입니다.**
강사 프로필과 자격 정보가 Schema.org Person JSON-LD로 자동 생성되어 AI 검색엔진의 권위 소스로 인용될 수 있습니다. 포즈별 HowTo·FAQPage·DefinedTerm JSON-LD가 API로 제공되어 Google 리치 결과 및 Perplexity 인용 확보가 가능합니다. 아티클 콘텐츠 관리와 스튜디오 정보 업데이트도 REST API로 지원합니다.

## 11page — 트러블슈팅

프로젝트 진행 중 마주쳤던 주요 문제들을 말씀드리겠습니다.

**첫 번째는 매칭 점수가 항상 0.0을 반환하는 문제입니다.**
프론트엔드의 목표값("Spinal_Mobility", "Back_Pain")과 DB의 태그값("back", "mobility")이 완전히 다른 어휘 체계를 사용하고 있었습니다. `equalsIgnoreCase` 비교는 단 한 번도 일치하지 않았습니다.
해결책으로 `GOAL_TAG_MAP`이라는 목표-태그 확장 매핑 테이블을 `MatchService`에 도입하여, UI 목표값을 DB 어휘로 자동 확장하는 방식을 적용했습니다. 결과적으로 점수가 정상 산출되었고, 상위 매칭 포즈가 올바르게 반환되었습니다.

**두 번째는 Kill-Switch 금기 포즈 차단 로직 구현 문제입니다.**
단순 조건 확인만으로는 severity 단계별 하드 차단이 되지 않았습니다.
Kill-Switch 개념을 도입하여 금기 포즈는 매칭 점수와 무관하게 `score=0, blocked=true`로 고정 반환하고, 차단 사유(예: "herniated disc contraindicated")를 응답에 함께 포함하도록 설계를 재정립했습니다.

**세 번째는 JSON-LD 직렬화 시 `@type` 필드가 누락되는 문제입니다.**
Jackson의 기본 직렬화 동작이 `Map<String, Object>` 내 null 필드를 예상과 다르게 처리하는 경우가 있었습니다.
`ObjectMapper`에 `NON_NULL` 직렬화 설정을 적용하고, JSON-LD 구조체를 `LinkedHashMap`으로 명시적으로 구성하는 방식으로 해결하였습니다.

**네 번째는 Streamlit 컨테이너 경로 설정으로 인한 404 문제입니다.**
`--server.baseUrlPath /safety` 옵션으로 Streamlit을 실행하면 루트 경로(`/`)에서 앱이 응답하지 않아 직접 포트 접근 시 404가 반환되었습니다.
`Caddyfile`에 `/safety*` → 8501 라우팅을 추가하고 컨테이너에서 baseUrlPath 옵션을 제거함으로써, Cloudflare 터널 경유(`match.yogaman.club/safety`)와 직접 포트 접근(`172.x.x.x:8501`) 모두 정상화되었습니다.

## 12page — 향후 개선 방향

마지막으로 서비스 완성도를 높이기 위한 개선 방향을 말씀드리겠습니다.

첫째, **예약 시스템 통합**입니다. Mindbody·1club API 연동을 통해 스튜디오 실시간 예약 기능을 추가하면, 매칭 추천이 예약 전환으로 직결되는 풀 파이프라인이 완성됩니다.

둘째, **AI 인용 범위 확대**입니다. 현재 주요 포즈에만 적용된 FAQPage·HowTo JSON-LD를 전체 2,700개 포즈와 아티클 콘텐츠까지 확장하면, Perplexity·Gemini 등 AI 검색엔진에서의 인용 빈도를 크게 높일 수 있습니다.

셋째, **수련 이력 기반 개인화**입니다. 사용자의 세션 이력을 축적하여 점진적 난이도 조절과 목표 추적 기능을 제공하면 재방문율과 서비스 완성도를 함께 높일 수 있습니다.

넷째, **모바일 앱 전환**입니다. 현재 웹 기반 서비스를 React Native로 확장하면 더 넓은 사용자층에 도달할 수 있습니다.

이와 같은 개선을 통해 단순 포즈 추천을 넘어, **요가 수련의 전 여정을 아우르는 신뢰 기반 AI 매칭 플랫폼**으로 발전시킬 수 있을 것이라 생각합니다.

## 시연 Flow

### [수련자 — 매칭 시연]
서비스 접속 (`match.yogaman.club`)
→ 신체 조건 입력: *"허리 디스크 있고, 스트레스 해소 목적, 30분 초보자"*
→ Kill-Switch 필터 동작 확인 (차단 포즈 `blocked: true` 표시)
→ 매칭 점수 결과 상위 5개 확인 (score, benefits 태그 표시)
→ 상위 추천 포즈 상세 페이지 (`/api/v1/poses/{id}`)
→ 경험 레벨·가용 시간 필터 변경 → 결과 재정렬 확인

### [RAG 챗봇 시연]
챗봇 접속 (`match.yogaman.club/chat`)
→ 자유 질문: *"다운독 포즈 척추 효과 설명해줘"*
→ 벡터 검색 결과 + 인용 소스(책 페이지) 확인
→ 후속 질문: *"허리 디스크 있으면 어떤 포즈 피해야 해?"*
→ 안전 필터 기반 답변 확인

### [AEO/SEO JSON-LD 시연]
`GET /api/v1/poses/{id}/faq` → FAQPage JSON-LD 확인
→ `GET /api/v1/glossary/{poseId}` → DefinedTerm JSON-LD 확인
→ `GET /api/v1/instructors/{id}/jsonld` → Person JSON-LD 확인
→ `GET /api/v1/articles/yoga-for-lower-back-pain/jsonld` → Article JSON-LD 확인

### [안전 필터 데모 — Streamlit]
`match.yogaman.club/safety` 접속
→ 고위험 포즈 입력 → 킬스위치 차단 시각화 확인

*(시간이 남는다면 → 아티클 목록 `/api/v1/articles`, 스튜디오 위치 검색, OpenAPI 문서 `/swagger-ui.html`)*
